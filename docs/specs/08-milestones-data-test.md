# 08 里程碑、数据与测试规范

## 1. 里程碑

| 阶段 | 主线任务（决策系统） | 支线任务（数据与模型） | 预计耗时 |
|------|----------------------|------------------------|----------|
| 第一阶段 | 搭建 LangGraph + MCP 框架骨架 | 安装 LeRobot，录制 50-100 组分拣动作数据 | 2-3 周 |
| 第二阶段 | 实现各 Agent 节点与状态流转 | 使用录制数据微调 SmolVLA 模型 | 4-5 周 |
| 第三阶段 | 完成前端界面开发与 MCP 工具集成 | 将 SmolVLA 封装为 MCP 工具，测试推理 | 2-3 周 |
| 第四阶段 | 端到端闭环调试，撰写论文 | 系统稳定性优化，准备答辩演示 | 2-3 周 |

## 2. 数据采集方案

- 场景：桌面物体分拣
- 示例任务：不同颜色方块分类放入指定区域
- 采集方式：`LeRobot control_robot.py` 人工示教
- 数据量：50-100 个成功轨迹
- 数据覆盖：不同物体位置和抓取角度
- 数据格式：`Parquet + 视频帧`

## 3. 测试验证

### 3.1 单元测试

- 感知层各 `MCP` 工具独立调用测试
- 执行层各原子工具独立调用测试
- 配置与错误处理逻辑测试

### 3.2 集成测试

- `LangGraph` 图在模拟工具环境中的逻辑正确性
- 感知到执行的最小闭环流程测试
- 前后端配置联动测试
- 第三阶段前端最小工作台对 `bootstrap / config / tools / runs / snapshot / events` 合同的消费接线评估
- 前端 `npm run build` 构建门禁，确保最小工作台工程至少达到可构建、可交付静态产物的状态

### 3.3 实物测试

- 至少完成 10 次连续成功的分拣任务
- 记录成功率、失败原因和环境条件

## 4. 阶段交付要求

- 每一阶段结束时必须更新 `docs/records/MILESTONES.md`
- 每一阶段必须补充对应测试结果到 `docs/records/TEST_REPORT.md`
- 每一阶段必须同步检查论文进度与实验材料记录

## 5. 第二阶段完成边界

第二阶段主线完成以“节点与状态流转能力稳定”为准，不以后续真实硬件或前端联动是否完成作为唯一判断条件。

第二阶段完成所需事实：
- 决策层已完成 phase-2 主线增强，`task_planner -> scene_analyzer -> action_decider -> executor -> verifier` 的节点职责保持不变，但 capability/action 选择与状态流转比第一阶段更完整。
- 感知层与执行层已形成 phase-2 可继续演进的接口边界：统一 `ToolEnvelope`、结构化 `ee_pose`、provider/adapter 工厂、execution capability/action contract。
- 统一入口仍保持组合根角色，不承担跨层业务翻译，并能稳定装配 phase-2 当前所需能力。
- 回归测试已覆盖 phase-2 主线新增能力，并保持全量通过。

第二阶段完成但可继续并行未闭合的外部项：
- Ubuntu 实机安装与运行验证
- 前端联动与前后端配置联动
- 真实 `VLM` / `LLM` / `SmolVLA` / 机械臂链路
- 数据采集与微调支线的真实结果

约束：
- 上述外部项未完成时，可以表述为“第二阶段主线完成，外部验证项继续并行推进”，不得表述为“端到端完成”或“真实链路完成”。

## 6. 第三阶段前端当前验收口径

第三阶段前端当前需要区分三层事实，避免把“代码存在”误写成“联调完成”：

- 构建门禁：`frontend/` 必须通过 `npm run build`
- 代码接线：前端最小工作台必须对齐 `docs/specs/07-frontend-spec.md` 中 `FE-A01 ~ FE-A09`
- 浏览器留痕：只有在真实页面完成初始化、run 提交、事件追更与终态收口，并留下人工记录或自动化证据后，才能写成“联调通过”

当前第三阶段允许写入 `TEST_REPORT.md` 的通过表述为：

- “前端最小联调骨架构建通过”
- “前端最小联调骨架代码接线通过”

当前第三阶段不允许写入的表述为：

- “真实前后端联调完成”
- “生产级流式能力完成”
- “端到端闭环完成”

# 03 系统架构

## 1. 架构目标

本项目采用“感知-决策-执行”三层解耦架构，通过 `MCP` 连接各层能力，并由前端提供任务输入、模型配置和状态展示能力。

## 2. 分层结构

### 2.1 前端界面

负责：
- 模型选择
- 任务输入
- 状态监控
- MCP 工具扩展面板

边界：
- 只负责交互、配置和展示
- 不承担核心业务决策逻辑

### 2.2 决策层

技术基线：
- `LangGraph + MCP Client`

负责：
- 多 Agent 节点编排
- 状态流转
- 条件分支
- 闭环控制

边界：
- 不直接操作硬件
- 不承担感知或执行层内部实现

### 2.3 感知层

技术基线：
- `MCP Server + VLM`

负责：
- 图像采集
- 机器人状态获取
- 场景描述生成

边界：
- 对外只暴露标准化工具
- 不与前端或执行层直接耦合内部实现

### 2.4 执行层

技术基线：
- `MCP Server + SmolVLA + LeRobot`

负责：
- 原子动作执行
- 高层抓取技能执行
- 机械臂动作下发

边界：
- `SmolVLA` 为固定核心技能
- 新能力通过新增 `MCP` 工具扩展，不直接替换核心技能

### 2.5 统一入口与装配层

技术基线：
- 统一启动入口 + 组合根装配

负责：
- 加载配置
- 选择运行模式
- 装配 `decision`、`perception`、`execution`
- 注入统一 `MCP gateway/client`
- 暴露启动入口

边界：
- 只负责装配和运行时切换
- 不承担跨层工具语义翻译
- 不承担结果归一化或状态协议设计
- 不直接实现感知、决策、执行业务逻辑

## 3. 主流程

固定主流程：
1. 用户输入自然语言指令。
2. 决策层生成任务序列。
3. 决策层调用感知层获取场景信息。
4. 决策层调用执行层产生并执行动作。
5. 决策层再次验证任务状态。
6. 根据验证结果继续闭环或结束任务。

## 4. 架构约束

- 层间通信通过 `MCP` 或显式适配层完成。
- 统一入口必须保持组合根角色，只负责配置、装配和运行模式切换。
- 决策层不直接访问硬件。
- 决策层通过稳定 capability contract 选择动作，不依赖感知层或执行层内部实现。
- 前端不直接承担核心业务决策。
- 感知层只输出感知合同，不输出动作建议或任务拆解结果。
- 执行层只负责动作执行、安全链和结果回传，不承担任务编排或闭环策略。
- 感知层和执行层对外只暴露标准工具，不泄露内部实现细节。
- 第二阶段主线只允许层内替换实现，不通过入口层新增跨层翻译逻辑。

## 5. 扩展原则

允许扩展：
- 新感知工具
- 新执行技能工具
- 新模型供应商适配器
- 新前端展示模块

扩展要求：
- 优先通过 `MCP` 做增量扩展。
- 不破坏既有主流程和分层边界。
- 第二阶段扩展优先在各层内部完成，不新增第四层业务中介。
- 若扩展能力改变层间合同，必须先更新 `03/04/05/06` 对应 `spec` 再落实现。
- 扩展能力后必须同步更新对应 `spec` 和测试记录。

# 03 系统架构

## 1. 架构目标

本项目采用“感知-决策-执行”三层解耦架构，通过 `MCP` 连接各层能力，并由前端提供任务输入、模型配置和状态展示能力。

## 2. 分层结构

### 2.1 前端界面

负责：
- 模型选择
- 任务输入
- 状态监控
- MCP 工具扩展面板

边界：
- 只负责交互、配置和展示
- 不承担核心业务决策逻辑

### 2.2 决策层

技术基线：
- `LangGraph + MCP Client`

负责：
- 多 Agent 节点编排
- 状态流转
- 条件分支
- 闭环控制

边界：
- 不直接操作硬件
- 不承担感知或执行层内部实现

### 2.3 感知层

技术基线：
- `MCP Server + VLM`

负责：
- 图像采集
- 机器人状态获取
- 场景描述生成

边界：
- 对外只暴露标准化工具
- 不与前端或执行层直接耦合内部实现

### 2.4 执行层

技术基线：
- `MCP Server + SmolVLA + LeRobot`

负责：
- 原子动作执行
- 高层抓取技能执行
- 机械臂动作下发

边界：
- `SmolVLA` 为固定核心技能
- 新能力通过新增 `MCP` 工具扩展，不直接替换核心技能

## 3. 主流程

固定主流程：
1. 用户输入自然语言指令。
2. 决策层生成任务序列。
3. 决策层调用感知层获取场景信息。
4. 决策层调用执行层产生并执行动作。
5. 决策层再次验证任务状态。
6. 根据验证结果继续闭环或结束任务。

## 4. 架构约束

- 层间通信通过 `MCP` 或显式适配层完成。
- 决策层不直接访问硬件。
- 前端不直接承担核心业务决策。
- 感知层和执行层对外只暴露标准工具，不泄露内部实现细节。

## 5. 扩展原则

允许扩展：
- 新感知工具
- 新执行技能工具
- 新模型供应商适配器
- 新前端展示模块

扩展要求：
- 优先通过 `MCP` 做增量扩展。
- 不破坏既有主流程和分层边界。
- 扩展能力后必须同步更新对应 `spec` 和测试记录。

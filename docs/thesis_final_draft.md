# 桌面级具身智能机器人感知-决策-执行一体化原型系统设计与实现

## 摘要

随着具身智能研究由单一算法验证逐步转向系统级集成与实体执行验证，如何构建一套面向桌面级机器人场景、具有清晰分层结构、统一接口约束、运行过程可观测且实验材料可追溯的原型系统，已成为毕业设计与工程研究中的关键问题。针对现有机器人原型系统中普遍存在的模块耦合度高、前后端联调困难、模型输出难以直接约束到物理动作空间、运行留痕不足等问题，本文设计并实现了一套面向桌面级机械臂的“感知—决策—执行”三层解耦一体化原型系统。

本文以大语言模型（Large Language Model, LLM）和视觉语言模型（Vision-Language Model, VLM）为上层智能核心，以轻量视觉-语言-动作模型 SmolVLA 为高阶动作生成能力，以统一运行时门面、状态图编排和执行安全边界为工程支撑，完成了感知层、决策层和执行层的整体实现。在感知层，系统构建了真实摄像头与虚拟图像源兼容的视觉采集通道，建立了 `get_image`、`get_robot_state` 与 `describe_scene` 三类统一工具合同，并通过结构化 Prompt 约束 VLM 输出；在决策层，系统基于 LangGraph 建立显式状态机，对自然语言输入、环境感知、任务规划、动作执行、执行验证和终态收口进行统一编排，同时引入动作对齐规则与启发式回退策略；在执行层，系统实现了 `move_to`、`move_home`、`grasp`、`release`、`servo_rotate` 与 `run_smolvla` 等动作工具，建立了输入校验、执行前检查、遥测反馈与结果封套机制，并完成了 SmolVLA 实体执行链路接入。

在系统实现层面，本文进一步构建了以前端工作台和后端运行服务为核心的一体化交互环境。前端采用 React 19、TypeScript 和 Zustand 实现运行态可视化工作台，支持自然语言任务输入、视频流显示、运行阶段展示、事件日志订阅和终态回显；后端采用最小 WSGI HTTP 服务与 Server-Sent Events（SSE）机制提供 `bootstrap`、`config`、`tools`、`runs`、`snapshot`、`events` 与 `video-stream` 等统一运行接口。实验结果表明，系统已完成 51 项后端单元与集成测试、前端 Playwright 端到端测试以及本地最小闭环 smoke 验证，形成了稳定的软件级回归基线。同时，围绕汉诺塔示教数据，本文完成了一轮 SmolVLA 训练实验，训练数据规模为 90 条 episode、33801 帧，训练总步数为 10000 步，最终 loss 为 0.042，最优 loss 为 0.038，并生成训练日志、checkpoint 记录及五张训练分析图。

综上，本文完成了一套具备感知、决策、执行闭环能力的桌面级具身智能机器人原型系统，实现了从自然语言输入到动作执行、从运行事件流到训练实验材料的完整工程闭环，可为桌面级具身智能系统的论文撰写、答辩展示与工程复现实验提供直接支撑。

**关键词**：具身智能；桌面级机械臂；LangGraph；大语言模型；视觉语言模型；SmolVLA；状态机编排

## 第1章 绪论

### 1.1 研究背景

具身智能强调智能体认知能力与物理实体、外部环境以及动作反馈之间的紧密耦合关系。与纯文本对话系统相比，具身智能系统不仅需要理解自然语言任务，还必须处理来自图像、机器人状态、执行反馈等多源异构信息，并将高层语义映射为具有时序性、安全边界和验证机制的物理动作序列。近年来，大语言模型在自然语言理解、任务拆解和策略生成方面展现出显著能力，视觉语言模型在图像理解、关系抽取和场景描述方面具备较高实用价值，视觉-语言-动作模型则进一步尝试将图像与文本直接映射为连续动作输出。这些能力的发展为机器人系统的智能化提供了新的技术路径。

然而，将上述模型能力真正应用于机器人系统时，工程问题远比算法问题更为复杂。首先，感知、决策与执行环节通常由不同模块实现，若缺乏统一合同与清晰边界，系统极易演化为高度耦合的脚本式原型，难以维护与扩展。其次，大模型输出天然具有概率性与开放性，若缺乏结构化约束与动作空间限制，模型规划结果可能偏离物理可执行边界。再次，传统机器人调试流程往往依赖命令行与零散日志，难以同时观察任务输入、当前配置、视觉结果、规划状态和执行回传，不利于系统验证与论文留痕。对于毕业设计而言，仅有若干可运行脚本并不足以支撑完整论文，必须同步形成结构化代码实现、测试证据、运行截图、配置模板、训练结果与图表材料。

桌面级机械臂平台为上述问题提供了较为适合的研究载体。一方面，桌面级平台部署成本相对较低，实验环境可控，便于持续联调与多轮验证；另一方面，其任务类型具有明确边界，如场景观察、单舵机控制、抓取放置、顺序操作与汉诺塔类任务，适合围绕“感知—决策—执行”闭环开展系统化设计。因此，围绕桌面级机械臂构建一套具备统一架构、统一状态流和统一实验材料出口的具身智能原型系统，具有明确的研究意义和实践价值。

### 1.2 研究意义

本文的研究意义主要体现在以下三个方面。

第一，从系统工程视角组织模型能力。当前许多研究更关注单一模型在感知、规划或动作生成中的效果，而本文将关注重点放在如何把感知模型、规划模型、执行模型、前端工作台和后端运行时组织为一个可运行、可验证、可回放的统一系统。该视角更接近真实机器人系统的开发需求，也更适合毕业设计对完整性和可复现性的要求。

第二，从工程约束视角保障模型可控性。本文并未直接让模型自由生成任意动作，而是通过结构化 Prompt、动作对齐规则、显式状态图与执行安全边界，将模型输出压缩到可验证、可执行、可回传的动作空间内。该方法为大模型参与机器人决策时如何兼顾灵活性与安全性提供了可操作的实现路径。

第三，从论文材料视角构建可追溯证据链。本文在系统开发过程中同步沉淀运行截图、测试报告、Prompt 资产、架构图、训练日志、训练曲线与数据统计结果，使系统实现过程与论文写作过程保持一致。这种做法使论文不仅能够描述“做了什么”，还能直接展示“如何验证”和“如何留痕”。

### 1.3 国内外研究现状

在机器人任务规划方向，Google 提出的 SayCan、RT-1、RT-2 等工作显示，语言理解能力与视觉理解能力可以显著增强机器人在开放场景中的任务选择与动作组织能力。其中，SayCan 强调语言策略与可执行技能集合之间的关联，RT 系列模型则探索了以多模态大模型为核心的机器人任务建模方式。在空间语义与规划控制方向，Stanford 的 VoxPoser 等工作进一步说明，语言模型不仅能够处理抽象语义，还能够参与空间目标和物理关系的推理过程。

在系统编排与多智能体工作流方向，LangChain 及其发展出的 LangGraph 为构建显式状态图、条件路由和多节点推理流提供了较成熟的工程支撑。与简单的链式调用相比，状态图方法更适合处理机器人系统中普遍存在的“感知—规划—执行—验证—补偿”闭环流程。对于需要在不同阶段调用不同工具、处理不同状态并记录事件的具身智能系统而言，状态图式编排具有较高适配性。

在视觉-语言-动作模型方向，SmolVLA 代表了轻量化 VLA 模型的一类实现思路，其目标是在更紧凑的参数规模下完成从视觉输入和文本任务描述到动作输出的映射。与大规模 VLA 模型相比，轻量化模型更适合桌面级实验平台的快速迭代、训练调试和本地部署。

尽管上述研究分别在模型能力、规划机制和训练方法上取得了进展，但从毕业设计的工程实践角度看，仍然缺少一套同时覆盖前端工作台、后端运行时、统一工具合同、状态图编排、实体执行接入和训练材料整理的完整样例。本文的工作正是面向这一空缺展开，力图把模型、系统与论文材料三者统一到一套连续工程框架中。

### 1.4 研究内容与论文结构

围绕上述问题，本文完成了以下研究内容：构建感知层、决策层与执行层三层解耦架构；建立统一运行时装配和前后端门面接口；完成基于 LangGraph 的状态图编排；完成结构化 Prompt 资产和动作对齐机制；完成执行层安全闭环和 SmolVLA 实体执行接入；完成前端工作台与后端事件流联调；完成自动化测试、运行留痕和训练实验材料整理。

全文共分为九章。第1章介绍研究背景、研究意义和相关现状；第2章讨论系统所依赖的理论基础与关键技术；第3章给出系统总体架构与运行流程；第4章至第6章分别讨论感知层、决策层和执行层的设计与实现；第7章介绍一体化工作台与后端服务实现；第8章给出系统测试、运行证据与训练实验结果分析；第9章对全文工作进行总结。

## 第2章 相关技术与理论基础

### 2.1 具身智能系统的分层组成

从系统结构角度看，具身智能系统通常可以抽象为感知层、决策层和执行层三个基本组成部分。感知层负责从环境与机器人本体采集原始信息，并将其组织为可被上层消费的结构化语义结果；决策层负责将用户任务、感知结果与历史上下文整合为具体动作选择；执行层则负责完成动作下发、硬件控制、执行验证与状态回传。

在实际工程中，这三个层次往往不是天然分离的。若视觉识别、自然语言解析和机械臂控制逻辑被混写在一个脚本中，则系统虽可在短期内完成演示，但极难支持持续测试、模块替换和论文级材料沉淀。本文因此将分层设计视为系统实现的首要原则，通过统一合同和统一运行时将三层能力装配为一个整体。

### 2.2 大语言模型与视觉语言模型

大语言模型在本系统中的主要作用是进行自然语言理解、任务拆解、能力选择与动作规划。用户在桌面级场景下常以自然语言方式描述目标，例如“描述当前画面”“让 1 号舵机左转 15 度”“执行抓取并归位”等，这些输入若直接转换为底层参数命令，会导致系统入口过于僵硬。LLM 能够在保留自然语言交互形式的同时，输出结构化任务信息，从而成为决策层的重要组成部分。

视觉语言模型在本系统中的作用是完成对图像内容的结构化描述。与纯视觉检测模型相比，VLM 更擅长输出包含物体类别、相对位置、遮挡关系和风险提示的语言化解释，这有助于让决策层在统一语义接口上处理环境信息。本文通过 Prompt 约束与 JSON 提取逻辑，使 VLM 输出能够稳定映射为 `scene_description`、`confidence` 和 `structured_observations` 等字段，进而与状态图节点形成闭环关系。

### 2.3 LangGraph 与显式状态图编排

LangGraph 提供了一种基于显式节点与条件边的状态图编排方式，适合描述多阶段、可回退、可验证的机器人任务流程。与线性链式调用相比，状态图允许系统在不同节点调用不同工具、处理不同输入，并在阶段失败或置信度不足时跳转至相应补偿路径。

本文将 `trigger`、`nlu`、`sensory`、`assessment`、`active_perception`、`task_planning`、`pre_feedback`、`motion_control`、`verification`、`error_diagnosis`、`compensation`、`goal_check` 和 `final_status` 等过程组织成统一状态图，使系统能够明确表达每一步的语义角色与状态依赖关系。这种设计使运行过程可解释、可追踪，也便于在前端页面中进行可视化展示。

### 2.4 SmolVLA 与轻量视觉-语言-动作模型

SmolVLA 属于轻量化视觉-语言-动作模型，其核心思想是在统一模型内部融合视觉输入、文本任务描述与动作生成能力，从而在更紧凑的模型规模下支持从图像与任务到动作序列的映射。在桌面级实验场景中，SmolVLA 的优势在于便于本地训练、快速验证与系统接入。

本文在执行层中将 `run_smolvla` 设计为高阶技能接口，使其能够接收任务描述、当前图像与机器人状态，并在执行层内部完成动作组织与结果回传。结合训练实验材料，SmolVLA 在本文中不仅体现为可调用的执行能力，同时也是一项具有可追溯训练过程和实验图表支撑的研究对象。

### 2.5 Prompt 工程与结构化输出约束

Prompt 工程在本系统中并非辅助性的文字说明，而是实现模型可控输出的重要工程机制。感知层 Prompt 约束 VLM 返回结构化 JSON；决策层 Prompt 约束 LLM 返回能力选择、动作选择、动作参数与面向用户的回复；汉诺塔技能 Prompt 则限定复杂顺序任务的规则边界与输出格式。

通过这种结构化 Prompt 设计，模型输出不再是任意文本，而是被纳入统一合同约束之中。结合规则对齐机制与执行层安全边界，Prompt 工程成为系统从“自然语言输入”过渡到“安全动作执行”的第一道约束机制。

## 第3章 系统总体设计与架构

### 3.1 系统设计目标

本文系统的设计目标可以概括为四个方面：低耦合、可替换、可观测、可验证。低耦合意味着感知、决策与执行各层具有明确职责边界；可替换意味着不同 provider、适配器与后端实现可以在统一合同下切换；可观测意味着系统运行过程能够通过前端工作台和后端事件流被持续观察；可验证意味着系统实现必须伴随测试报告、截图留痕、训练材料和图表产物。

### 3.2 三层解耦总体架构

系统总体采用感知层、决策层、执行层三层解耦结构，在其外部设置统一运行时装配层和前后端门面层。感知层负责图像采集、机器人状态读取与场景描述；决策层负责自然语言理解、任务规划、动作选择与状态流编排；执行层负责动作工具注册、安全校验、适配器下发与执行结果封套；统一运行时装配层负责读取配置、构建三层对象并完成注册；前后端门面层负责对页面暴露统一运行接口。

[图 3-1 系统总体架构图]

### 3.3 统一运行时装配机制

系统以统一配置对象为装配入口。启动时，配置首先被解析为统一 `AppConfig`，随后分别用于构建感知服务、执行服务和决策引擎。感知服务根据摄像头后端、机器人状态后端和 VLM provider 完成装配；执行服务根据机器人适配器、SmolVLA 后端和安全参数完成装配；决策服务根据 LLM provider 和统一 MCP client 构建状态图执行环境。

这种装配方式使各层之间的依赖关系由配置和运行时门面控制，而不是直接在业务逻辑中硬编码。由此，前端始终通过 `bootstrap`、`config`、`tools`、`runs`、`snapshot` 和 `events` 等统一接口消费运行时状态，不必了解底层具体实现细节。

### 3.4 合同化接口设计

本文系统在层间通信上不采用自由格式字典，而采用能力合同与结构化结果对象。感知层输出 `CapturedImage`、`RobotStateSnapshot` 与 `SceneDescriptionResult`；决策层输出包括 `selected_capability`、`selected_action`、`selected_action_args`、`assistant_response` 在内的结构化规划结果；执行层输出 `ExecutionToolResult`，其中包含动作名称、状态、消息、日志、最新机器人状态与附加元数据。

合同化接口设计直接降低了模块之间的语义歧义。一旦合同字段确定，模块内部实现可以演化，但外部消费逻辑无需同步重写。这是系统能够同时容纳真实 provider、mock fallback、视频流展示、事件流回放和训练材料留痕的基础。

### 3.5 系统主流程

系统主流程如下：用户首先在前端工作台输入自然语言任务，前端通过 `POST /api/v1/runtime/runs` 创建一次新的运行任务；决策层进入 `trigger` 与 `nlu` 阶段，对输入进行解析；随后系统进入 `sensory` 阶段，从感知层获取图像、机器人状态和场景描述；若感知置信度不足，则进入主动感知流程；若置信度满足要求，则进入 `task_planning` 生成当前任务的能力选择与动作选择；之后系统进入 `pre_feedback` 和 `motion_control` 阶段完成执行前反馈和动作下发；执行结束后进入 `verification` 节点，根据执行结果决定是否收口；最终系统进入 `final_status`，输出终态结果并关闭事件流。

在这一过程中，所有阶段信息、快照信息、事件版本和运行终态都通过 `run_id` 生命周期被统一维护，并通过 SSE 与快照接口实时推送到前端，从而构成完整的运行闭环。

## 第4章 环境感知层设计与实现

### 4.1 感知层职责与输入来源

环境感知层负责完成三类核心输入的组织：当前图像、机器人状态和场景语义。图像输入来自摄像头适配器，机器人状态来自机器人状态后端，场景语义来自 VLM 描述模块。三者共同构成系统在任务执行前的环境认知基础。

在本文实现中，感知层既支持真实摄像头，也支持本地测试图像；既支持真实机器人状态回传，也支持 mock 状态来源；既支持通过远程兼容接口调用真实 VLM，也支持测试环境中的降级返回。通过这种组织方式，感知层既满足实体联调需求，又满足软件级测试需求。

### 4.2 感知工具合同设计

为降低上层模块对感知层内部实现的依赖，本文将感知能力封装为三个统一工具接口：`get_image`、`get_robot_state` 和 `describe_scene`。其中，`get_image` 返回图像 base64 编码、时间戳和尺寸信息；`get_robot_state` 返回关节角、末端位姿和夹爪状态；`describe_scene` 返回包含自然语言描述、置信度和结构化观察结果的统一对象。

合同化设计使决策层只需关注字段语义，而无需关注图像来自何种相机、状态来自何种后端或视觉模型来自何种 provider。这种解耦能力是系统跨环境运行与前后端稳定接线的前提。

### 4.3 VLM 结构化输出机制

在实现中，VLM 并不直接输出自由叙述文本，而是在 Prompt 约束下输出结构化 JSON。系统要求视觉模型至少返回 `scene_description`、`confidence` 和 `structured_observations` 三类字段，其中 `structured_observations` 下进一步包含 `objects`、`relations`、`robot_grasp_state` 和 `risk_flags`。通过这一机制，感知层输出能够直接进入决策层状态流，不需要额外的人为文本清洗。

对于不同 provider 的返回差异，系统内部实现了统一提取与归一化处理逻辑。无论底层来自 OpenAI 兼容接口、Anthropic 兼容接口还是 Ollama 兼容接口，最终都被转换为统一 `SceneDescriptionResult` 结构，从而保证上层消费逻辑稳定。

### 4.4 主动感知与置信度评估

针对桌面级场景中常见的遮挡、模糊和观察角度不足问题，本文在感知与决策交界处引入置信度评估机制。系统在 `assessment` 节点根据 VLM 输出的 `confidence` 值判断当前感知结果是否满足规划条件；当置信度低于阈值时，系统进入 `active_perception` 分支并重新获取环境信息，以提升规划质量。

该机制的意义在于，系统并不把单次视觉输出视为绝对正确，而是显式把“是否足够可信”纳入状态流判断。由此，主动感知成为感知层对决策层输出质量负责的一种工程机制。

### 4.5 感知层运行证据

本文已完成真实摄像头视频流在前端工作台中的展示接线，页面左上角能够持续显示 MJPEG 视频流，并与当前任务快照和感知结果同步更新。该运行证据说明感知层不仅存在于接口定义中，而且已经在前端运行环境中形成直观展示。配合 `describe_scene` 返回内容，系统构成了从图像采集到结构化场景描述的完整感知闭环。

## 第5章 决策规划层设计与实现

### 5.1 决策层总体设计思路

决策层是本文系统中的核心调度中枢，其职责是将用户输入、感知结果和执行反馈组织为一条可追踪、可验证、可回放的状态流。本文并未把决策简化为单次模型调用，而是采用“状态图编排 + 结构化规划 + 规则约束 + 结果验证”的复合策略，从而保证任务规划既具有灵活性，又保留工程可控性。

### 5.2 LangGraph 状态流转机制

基于 LangGraph，系统构建了包含触发、解析、感知、评估、主动感知、规划、执行前反馈、动作控制、执行验证、错误诊断、补偿控制、目标检查和终态收口等阶段的统一状态图。每个节点承担单一职责，节点之间通过显式边和条件路由构成任务流转关系。

[图 5-1 决策层 LangGraph 状态流图]

这种显式状态图结构具有两方面优势。其一，系统行为可解释，开发者能够明确看到当前任务位于哪一阶段；其二，系统运行可回放，前端能够根据事件流恢复整个运行序列。这一设计为前端工作台的执行流可视化提供了直接支撑。

### 5.3 任务规划与动作选择机制

当系统进入 `task_planning` 节点时，决策层综合用户原始指令、当前任务片段、场景描述、机器人状态和历史反馈，调用决策 provider 生成结构化规划结果。规划输出被约束为 JSON 结构，包含能力选择、动作选择、动作参数、原因说明以及面向用户的简短回复。

在动作层面，系统已支持将任务映射为 `describe_scene`、`move_home`、`servo_rotate`、`release` 或 `run_smolvla` 等动作空间。这一映射机制使系统既能处理视觉问答类任务，也能处理舵机控制、归位和抓取执行类任务。

### 5.4 动作对齐规则与启发式回退

为保证模型规划结果与任务语义保持一致，系统在决策层中加入了显式动作对齐规则。例如，单舵机旋转任务只能选择 `servo_rotate`，归位任务只能选择 `move_home`，场景问答任务只能选择 `describe_scene`。若模型输出与这些规则不一致，系统会自动应用对齐覆盖，并在元数据中记录覆盖原因。

除规则对齐外，系统还实现了启发式回退机制。当模型响应超时、格式错误或结果不可解析时，系统可根据任务关键词与当前上下文采用规则化方式生成最低可用动作方案。该机制使决策层具有更好的工程鲁棒性，也使自动化测试能够覆盖异常路径。

### 5.5 Prompt 驱动的结构化规划

决策层核心 Prompt 资产 `DECISION_PLANNING_SYSTEM_PROMPT` 要求模型只返回规定字段的 JSON 结果，并通过动作空间限制降低越权生成风险。与单纯把 Prompt 视为说明文本不同，本文把 Prompt 工程视为决策可控性的组成部分。它与状态图、对齐规则和执行安全边界共同构成多级约束链。

### 5.6 汉诺塔任务 skill 扩展示例

为验证系统对复杂顺序任务的表达能力，本文在项目中新增了汉诺塔任务 skill。该 skill 能够依据“一次只能移动一个圆环”“大圆环不能放在小圆环之上”等规则，递归生成合法步骤序列，并输出可追踪的原子任务提示。对应实现已包含专用 Prompt、skill 模块、规范文档和测试文件，说明系统不仅支持单步动作调用，也支持较长序列任务的知识化扩展。

## 第6章 动作执行层设计与实现

### 6.1 执行层职责与工具集合

执行层负责将决策层生成的动作选择落实为可执行工具调用，并在执行完成后回传标准化结果。当前系统中，执行层已注册 `move_to`、`move_home`、`grasp`、`release`、`servo_rotate`、`clear_emergency_stop` 和 `run_smolvla` 等动作工具，覆盖了原子动作控制、高阶技能调用和安全状态管理等场景。

### 6.2 原子动作与高阶技能协同

在本文设计中，`move_to`、`move_home`、`grasp`、`release` 和 `servo_rotate` 可视为原子或近原子动作，它们具有参数明确、动作边界清晰、便于验证与回放等特点。`run_smolvla` 则属于高阶技能接口，其输入同时包含任务描述、图像和机器人状态，适合用于表达抓取、放置等需要联合视觉和动作推理的任务。

通过同时保留原子动作与高阶技能，执行层既能支持明确控制型任务，也能支持基于模型能力的复杂动作生成任务，从而形成较完整的动作空间。

### 6.3 安全边界与执行前检查

具身系统中的动作执行必须满足物理安全约束。为此，本文在 `SafetyManager` 中实现了输入校验、执行前检查和遥测验证三类安全策略。输入校验负责检查参数类型和范围；执行前检查负责约束单步位移距离、姿态边界和奇异区；遥测验证则依据执行后的状态回传确认动作是否处于安全区间。

以位移检查为例，系统在执行前计算目标位置与当前位置之间的欧氏距离：

$$
d = \sqrt{(x_t-x_c)^2 + (y_t-y_c)^2 + (z_t-z_c)^2}
$$

当位移大于设定阈值时，系统将拒绝当前动作调用并返回 `SafetyError`。该机制有效避免了由于上层规划异常导致的超范围动作。

### 6.4 遥测反馈与结果封套

执行层并不只返回简单的成功或失败标记，而是返回包含动作名称、状态、消息、日志、机器人状态和附加元数据的结构化结果对象。该对象一方面为决策层执行验证提供依据，另一方面为前端工作台展示执行回传提供标准载荷。

在当前实现中，前端快照已能够展示 `last_execution`、`execution_feedback` 和 `robot_state` 等信息，说明执行层结果不仅在后端内部消费，而且已完成页面级联调。

### 6.5 SmolVLA 实体执行链路

本文完成了 `run_smolvla` 的实体执行链路接入，使其不再只是接口占位，而成为执行层中具备实际动作生成与下发能力的高阶技能工具。该事实使执行层从“具有扩展设计”进入“具备实体执行能力”的阶段。结合 `move_home`、`servo_rotate` 等工具的接线与运行结果，执行层已形成“工具注册—安全校验—适配器下发—遥测回传—结果封套”的完整闭环。

## 第7章 一体化工作台与后端服务实现

### 7.1 前端工作台设计目标

前端工作台是本文系统的主要交互入口和可观测性载体。其目标并非仅仅提供一个输入框，而是同时承担自然语言任务提交、配置展示、运行阶段可视化、视频流查看、事件日志回放和终态结果呈现等功能。通过统一工作台，系统运行过程由“后台脚本行为”转化为“可被观察和分析的运行界面”。

### 7.2 后端 HTTP、SSE 与视频流服务

为支撑上述工作台，后端提供了最小但完整的运行时服务接口。其中，`bootstrap` 用于页面初始化，`config` 用于读取与更新配置，`tools` 用于获取工具定义，`runs` 用于启动任务并返回 `run_id`、`snapshot_url` 和 `events_url`，`events` 用于返回阶段事件，`video-stream` 用于以 MJPEG 形式输出当前视觉流。

SSE 机制使后端能够在任务运行过程中持续向前端推送阶段状态和事件信息，从而使前端不必频繁轮询即可获取运行进度。MJPEG 视频流则为感知层运行结果提供直观载体。

### 7.3 前端状态管理与接口接线

前端采用 Zustand 统一管理配置草稿、工具集合、当前任务、运行接受结果、快照数据、事件列表和运行状态。页面初始化时，前端并行请求 `bootstrap`、`config` 和 `tools`；在提交任务后，前端依据返回的 `events_url` 建立 SSE 订阅，并通过 `snapshot_url` 进行必要的状态补偿同步。

这种“事件流 + 快照”的双重接线方式提高了页面展示的完整性与稳定性，也使系统在任务过快完成或事件顺序变化时仍能回显正确终态。

### 7.4 `run_id` 生命周期管理

系统中每一次运行任务均对应唯一 `run_id`。后端通过 `RunRegistry` 为其维护事件序列、最新快照、终态标志、版本号和停止请求状态；前端则根据 `run_id` 持续更新当前运行视图。当任务进入终态后，前端关闭对应事件流，从而完成一次完整生命周期管理。

`run_id` 机制不仅是运行控制的基础，也是论文中时序图、截图说明和事件分析的重要组织线索。

本文工作表明，在桌面级机械臂场景下，通过统一合同、显式状态图和前后端一体化工作台，可以有效组织 LLM、VLM 与 SmolVLA 等多类模型能力，使其在同一系统中完成感知、规划、执行与验证的连续闭环。同时，通过同步沉淀测试报告、运行截图、Prompt 资产、训练日志和图表材料，系统开发过程能够自然转化为论文级证据链。本文形成的原型系统与终稿材料已经具备较完整的研究展示价值和工程复现实用价值。

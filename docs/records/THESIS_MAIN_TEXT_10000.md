# 桌面级具身智能机器人感知-决策-执行一体化原型系统设计与实现

## 第1章 绪论

### 1.1 研究背景

近年来，人工智能技术的发展正在推动机器人系统由传统的规则控制、脚本控制逐步向具备环境理解、任务规划与自主执行能力的具身智能系统演化。与工业流水线中高度结构化、重复性的固定任务不同，开放桌面场景中的机器人操作任务往往具有目标描述自然语言化、环境状态动态变化、视觉输入不稳定以及执行结果需要持续验证等特点。这类任务要求系统不仅能够“看见”目标，还要能够“理解”用户意图，并在动作执行后根据反馈进行修正。因此，感知、决策与执行的协同问题已经成为具身智能系统工程化落地的核心问题之一。

在学术研究中，大语言模型与视觉语言模型的快速发展为机器人系统提供了新的方法论基础。大语言模型具备较强的自然语言理解、任务拆解和策略推理能力，能够将用户输入的抽象任务目标转化为更加明确的子任务或动作意图；视觉语言模型则擅长对图像内容进行语义描述、目标识别和风险提示，能够为机器人系统提供接近人类表达形式的环境理解结果。这使得机器人系统不再局限于刚性的坐标控制，而有可能形成“看到环境、理解任务、规划步骤、执行动作、验证结果”的闭环式工作模式。

然而，大模型进入机器人链路后并不天然等于系统落地。首先，模型输出具有概率性和开放性，如果缺乏工程约束，很容易产生超出物理可执行边界的规划结果。其次，机器人系统天然涉及跨模态输入与跨模块协作，感知、规划和动作控制若紧耦合在同一逻辑中，后续将难以调试、替换和扩展。再次，很多开源项目在算法层面展示了较好的能力，但缺少一套完整的前后端联调平台，导致研究者难以观察系统内部状态流转，也难以积累规范化的实验材料。对于毕业设计而言，如果没有完整的架构与证据链支撑，仅凭若干脚本或单次演示，很难形成说服力较强的论文成果。

桌面级机械臂平台为上述问题的研究提供了较为理想的实验载体。一方面，桌面机械臂成本相对较低，部署空间要求小，适合个人课题环境下持续开展调试与验证；另一方面，桌面场景中的任务类型相对清晰，例如目标识别、单物体抓取、放置、分拣以及汉诺塔类顺序任务，便于围绕“感知-决策-执行”链路开展系统化研究。此外，桌面平台易于结合摄像头、前端工作台、示教数据采集和模型微调过程，为构建可复用的具身智能实验平台提供了良好基础。

基于上述背景，本文围绕“桌面级具身智能机器人感知-决策-执行一体化原型系统”的设计与实现展开研究，重点关注系统架构解耦、运行时状态可观测、模型接入可替换、任务执行可验证以及论文材料可追溯等工程问题，尝试给出一套适合毕业设计阶段落地的系统方案。

### 1.2 研究意义

本课题的研究意义主要体现在三个方面。

第一，课题强调从系统视角而不是单模型视角研究具身智能问题。当前不少工作集中于验证某一模型在问答、感知或动作预测上的效果，但在真实机器人系统中，模型能力只是整体系统的一部分。如何将模型能力可靠地组织到统一状态流中，如何让前端、后端、模型、相机和执行模块共同协作，如何形成完整的错误处理与验证机制，这些问题具有突出的系统工程意义。本文通过构建三层解耦架构和统一运行时门面，给出了完整的系统工程实现。

第二，课题针对大模型引入机器人链路后的“可控性”问题提出了结构化约束思路。通过在感知层引入结构化视觉输出合同，在决策层引入受限 JSON 规划提示词，在执行层引入安全前置检查、遥测反馈和终态验证机制，系统可以在保持大模型灵活性的同时，尽量减少自由生成所带来的越权规划风险。这种做法对于当前具身智能系统如何在开放任务描述与严格物理边界之间取得平衡具有现实意义。

第三，课题重视“可观测性”与“可复现实验材料”的建设。毕业设计并不仅仅是写出能跑的代码，更要形成完整、可回溯、可论证的过程证据。本文所实现的一体化工作台不仅负责前端输入与状态展示，还承担运行过程留痕、配置管理、快照同步和事件订阅的功能。结合测试报告、运行截图、配置模板和 Prompt 资产，系统已经形成可直接服务于论文撰写与答辩展示的连续材料支持。这种“边开发边沉淀论文材料”的方式，也有助于提高课题结果的完整性与学术表达质量。

### 1.3 研究目标与主要内容

本文的总体目标是面向桌面级机器人场景，设计并实现一套具有清晰分层、统一合同、可观测运行态和可扩展模型接入能力的具身智能原型系统。围绕这一总体目标，本文进一步细化出以下具体目标。

第一，构建感知层、决策层和执行层三层解耦结构，确保各层既能独立替换实现，又能通过统一能力合同完成协同。第二，建立基于 LangGraph 的有状态决策流，使系统能够在任务输入、感知、规划、动作执行、验证和终态汇报之间完成闭环推进。第三，搭建一体化前端工作台，支持自然语言任务输入、配置修改、工具刷新、运行事件可视化、视频流显示和终态截图留痕。第四，在项目中引入真实 LLM/VLM provider 的接入路径，同时保留 mock fallback，以兼顾多环境适配与可验证性。第五，围绕当前项目完成边界，形成测试报告、实验记录、Prompt 资产、架构图和章节材料分配文档，使开发结果能够直接服务于论文写作。

基于以上目标，本文主要完成了以下内容：完成统一运行时装配层与后端最小 HTTP/SSE 服务骨架设计；完成感知层相机、机器人状态与 VLM 描述能力适配；完成决策层 LangGraph 状态图、受限规划提示词与 heuristic fallback 逻辑；完成执行层原子动作、高阶技能、安全前置检查与遥测回传边界；完成前端工作台对配置、工具、运行任务、快照和事件流的真实接线；新增汉诺塔任务 skill 作为复杂顺序任务扩展示例；补齐论文所需的附录、图表、截图与证据索引材料。

本文各章节安排如下：第2章介绍本系统所依赖的相关技术与理论基础，包括具身智能系统的组成方式、大语言模型与视觉语言模型的功能定位、LangGraph 状态图编排思想以及 Prompt 工程在系统可控性中的作用；第3章讨论系统总体设计与架构，重点说明三层解耦结构、统一运行时装配以及前后端协同模式；第4章至第6章分别介绍感知层、决策层和执行层的设计与实现细节；第7章介绍一体化工作台与后端服务实现；第8章给出系统测试、运行留痕与训练实验分析；最后对全文进行总结。

## 第2章 相关技术与理论基础

### 2.1 具身智能系统的基本组成

具身智能强调智能体的认知能力必须与物理实体、外部环境以及动作反馈紧密耦合。与纯文本对话系统不同，机器人系统不仅要处理语言输入，还需要实时接收图像、状态、位置等多模态信息，并将规划结果映射为具有时序性和安全约束的执行动作。因此，具身智能系统通常可以抽象为感知、决策与执行三个基本层次。

感知层负责采集环境与机器人自身状态信息，并将低层数据转化为可被上层模块消费的结构化语义结果。决策层负责理解用户任务、维护系统状态、选择工具或动作，并根据环境反馈进行闭环判断。执行层则负责真正的动作下发、底层控制以及动作结果回传。三者缺一不可：如果没有感知，系统无法确定环境状态；如果没有决策，系统只能执行固定脚本；如果没有执行反馈，系统便无法验证目标是否真正完成。

在工程实现中，这三个层次的边界经常被打乱。例如，有些原型直接把视觉识别、路径规划和机械臂控制混写在同一脚本中；有些系统则把前端展示逻辑嵌入后端任务逻辑，导致一旦模型或硬件接口变化，整个系统都需要修改。本文认为，面向毕业设计和后续长期扩展的具身智能平台，应优先保证分层清晰、接口稳定和状态可追踪，而不是追求一次性堆砌大量耦合功能。

### 2.2 大语言模型与视觉语言模型

大语言模型的核心价值在于自然语言理解、抽象目标拆解、上下文推理和策略描述能力。在机器人系统中，用户往往不会用精确的笛卡尔坐标去描述任务，而是说“把这个拿到那边”“看看桌面上有什么”“让1号舵机左转15度”等接近日常交流的表达。大语言模型能够将这些自然语言输入映射为更具结构化的任务意图，从而成为决策层的关键组成部分。

视觉语言模型则承担了感知层中语义理解的职责。传统的图像处理方法往往需要预先定义识别对象类别、阈值或视觉模板，而在开放桌面场景中，用户更关心的是“画面里有什么”“哪个物体可抓取”“当前是否存在风险”“夹爪是否处于闭合状态”。VLM 能够直接从视觉输入中生成描述性文本或结构化对象列表，为决策层提供比原始图像更高层的语义支持。

不过，模型能力并不意味着可以直接用于机器人动作执行。LLM 与 VLM 的输出本质上仍具有概率性，如果直接让模型随意生成动作序列，系统很容易出现不符合物理约束、缺少必要参数或忽略安全边界的问题。因此，本文采取的策略并不是把模型作为唯一控制者，而是把模型嵌入到显式状态流和结构化合同中，使其输出必须符合既定字段、既定动作空间和既定验证机制。这种“模型受约束地参与决策”的方式更适合当前阶段的桌面级具身智能系统实现。

### 2.3 LangGraph 与状态图式决策编排

机器人任务与普通的单轮文本问答有本质差异。机器人任务通常包含多个阶段，并且阶段之间可能根据环境反馈发生分支。例如，系统需要先获取感知结果，再判断置信度是否足够；若不足则触发主动感知；如果规划失败则要进行 fallback；如果执行失败则要进行错误诊断和补偿控制。这样的流程天然适合通过显式状态图来建模。

LangGraph 提供了一种有状态、可分支、可循环的图式编排方式。相较于简单的线性链式调用，状态图能够显式表达节点、边、路由条件以及终态收口逻辑，非常适合机器人系统中的闭环控制场景。本文在决策层中采用 LangGraph 构建状态流，将触发、语义解析、环境感知、置信度评估、任务规划、执行前反馈、动作控制、执行验证、错误诊断、补偿控制、目标检查和终态报告等过程组织成统一图结构，从而使系统在运行时具备更强的透明性和可调试性。

### 2.4 Prompt 工程与结构化输出约束

Prompt 工程在本系统中并不是简单的“给模型一句说明”，而是控制模型输出边界、稳定系统行为的关键工程手段。感知层通过系统提示词约束 VLM 只返回包含 `scene_description`、`confidence` 和 `structured_observations` 的结构化 JSON，从而降低感知输出在格式上的不确定性。决策层则通过受限规划提示词约束 LLM 只返回 `selected_capability`、`selected_action`、`selected_action_args`、`assistant_response` 等字段，并对舵机控制、归位、场景查询等任务给出显式动作空间限制。

这种设计的意义在于：一方面，模型输出更容易被程序消费；另一方面，即使模型本身具有较大自由度，也不会轻易突破系统所定义的动作边界。下文会结合具体 Prompt 资产说明这一机制如何在项目中落地。

### 2.5 SmolVLA 与轻量视觉-语言-动作模型

SmolVLA 代表了一类轻量化视觉-语言-动作模型，它尝试直接根据视觉输入和任务描述生成动作计划或动作序列。对于开放桌面场景而言，VLA 模型为复杂抓取和放置任务提供了统一接口的可能性。本文在执行层中保留了 `run_smolvla` 能力位，并将其作为高阶技能接口的一部分完成了实体执行接入。

结合当前项目进展，SmolVLA 已经不再只是接口预留或 mock 规划能力，而是已经完成实体执行链路打通，可以在真实执行场景下参与动作生成与下发，并且已经形成训练报告、训练曲线和日志材料。

## 第3章 系统总体设计与架构

### 3.1 系统设计目标

本系统的总体设计目标可以概括为“低耦合、可替换、可观测、可验证”。所谓低耦合，是指感知、决策与执行各层应具有明确职责边界，模块内部实现可随需求演进，而不影响其他层对其的调用方式。所谓可替换，是指系统应允许真实模型、mock provider、本地模型或远程兼容网关在统一合同下切换使用。所谓可观测，是指系统运行过程必须对开发者透明，能够通过前端工作台看到当前配置、当前阶段、当前快照和历史事件。所谓可验证，则是指系统应提供足够的自动化测试、运行日志、截图证据和附录材料，使开发结果不仅能运行，也能被论文和答辩所论证。

基于上述目标，本文没有采用“直接串联若干 API 形成单次演示脚本”的方式，而是围绕统一运行时和前后端门面设计整体架构。这样做虽然在早期开发阶段会增加一些合同和状态管理成本，但能够显著提高系统联调、复杂任务组织和论文材料沉淀的效率。

### 3.2 三层解耦总体架构

系统总体上采用感知层、决策层、执行层三层解耦结构。感知层通过相机适配器、机器人状态适配器与 VLM provider 共同构成，负责环境图像采集、机器人状态读取和场景理解输出；决策层以 LangGraph 为主线组织状态流转，结合真实 LLM provider 与 heuristic fallback 实现自然语言任务理解、工具选择和闭环控制；执行层则通过原子动作工具、高阶技能工具和安全管理器完成动作下发、遥测校验和结果回传。

在三层结构之外，系统还引入统一运行时装配层与后端门面层。统一运行时装配层负责根据配置构建 perception、decision、execution 和 unified MCP client，并将其收口为统一的 `Phase1Runtime`。后端门面层则通过 `FrontendRuntimeFacade` 和 `BackendHTTPApp` 对前端暴露稳定的 `bootstrap / config / tools / runs / snapshot / events / video-stream` 接口，使前端不需要理解底层模型或硬件细节，只需消费统一合同即可。

### 3.3 统一运行时装配机制

系统的统一装配入口位于后端运行时构建逻辑中。配置文件首先被解析为统一的 `AppConfig`，随后分别用于构建感知服务、执行服务和决策引擎。感知服务内部根据 `camera_backend`、`robot_state_backend` 和 `vlm_provider` 选择合适的适配器与 provider；执行服务内部根据 `robot_adapter`、`smolvla_backend` 和安全参数构建执行工具与安全边界；决策层内部则根据 `llm_provider` 装配相应决策 provider，同时持有统一 MCP client，用于访问感知层与执行层工具。

通过这一装配过程，系统可以较为灵活地在 mock-first 与真实链路模板之间切换。例如，在本地联调阶段可以使用 mock execution 进行完整前后端验证；在真实摄像头接入时，只需要修改 perception 分区中的相机后端和设备路径；在真实机械臂与 SmolVLA 实体执行链路接入后，也可以通过 execution 分区切换适配器和安全参数，而不需要重写前端与决策层逻辑。

### 3.4 系统主流程

从运行时角度看，系统主流程包括以下几个核心步骤。首先，用户在前端工作台中输入自然语言任务指令，并由前端通过 `POST /api/v1/runtime/runs` 启动一次新的 `run_id` 生命周期。其次，决策层进入 `trigger` 和 `nlu` 阶段，对用户任务进行接收与拆解，并生成待执行任务队列。随后，系统进入 `sensory` 节点，从感知层获取图像、机器人状态与场景描述，再通过 `assessment` 节点判断当前感知结果是否达到置信度阈值。若置信度不足，则进入主动感知分支；若满足要求，则进入 `task_planning`，由真实 LLM provider 或 heuristic fallback 生成动作选择方案。

在得到当前任务的能力选择和动作选择后，系统会先通过 `pre_feedback` 生成执行前反馈，再进入 `motion_control` 调用执行层工具完成动作下发。执行结束后，`verification` 节点会基于执行结果进行验证。如果失败，则进入错误诊断和补偿路径；如果成功，则进入完成反馈、目标检查和终态报告。整个过程中的阶段事件、快照和终态信息会被写入 `RunRegistry`，并通过后端门面实时回传前端，从而形成完整的可观测闭环。

### 3.5 架构扩展原则

为了保证系统后续可持续演化，本文在架构上遵循以下扩展原则。第一，优先通过稳定合同扩展新能力，而不是在上层逻辑中直接写死下层实现细节。第二，若新增复杂任务类型，应优先设计成独立 skill 或子规划模块，并保持与现有状态流兼容。第三，前端应继续作为“观察与交互界面”，而不是承担跨层业务语义翻译。第四，所有新增能力都应尽可能伴随配置模板、测试、截图和说明文档更新，以保证论文材料和工程实现始终同步。

在这一原则下，本文后续新增的汉诺塔任务 skill 便不是直接改写现有执行链路，而是先作为任务技能资产进入项目，通过独立规范文档、系统 Prompt 和步骤生成逻辑证明系统具备长程任务扩展的工程能力。

## 第4章 环境感知层设计与实现

### 4.1 感知层职责与输入来源

感知层是整个系统与外部物理世界建立联系的第一层。它的职责不仅包括“采集图像”，还包括读取机器人自身状态、理解场景语义、输出结构化结果，并在必要时为决策层提供降级路径。本文的感知层主要由相机适配器、机器人状态适配器和视觉语言模型 provider 三部分构成。

相机适配器负责从真实设备或 mock 数据源中获取当前图像。当前项目已经支持基于 OpenCV 的相机后端，并可通过 `camera_device_id`、`camera_width`、`camera_height` 和 `camera_fps` 等参数进行配置。机器人状态适配器负责读取当前关节状态和末端执行器位姿信息，用于辅助决策层完成 proprioceptive 闭环判断。VLM provider 则根据当前配置选择对应模型，并结合统一 Prompt 约束对图像生成结构化描述。

### 4.2 感知工具合同设计

为了避免决策层依赖感知层内部实现细节，本文将感知能力封装为三个稳定工具接口：`get_image`、`get_robot_state` 和 `describe_scene`。其中，`get_image` 负责返回当前图像的 base64 编码以及相机元数据；`get_robot_state` 返回关节角、末端位姿和状态时间戳；`describe_scene` 则接收图像和可选 Prompt，返回包含 `scene_description`、`confidence`、`structured_observations` 的结构化语义结果。

这种合同化设计的优势在于，决策层并不需要知道图像是通过哪个设备采集的，也不需要知道底层视觉模型来自哪个 provider。它只需要在 `sensory` 节点按统一方式调用工具并读取字段即可。相应地，当系统切换到别的视觉模型或机器人状态读取后端时，只要保持合同字段不变，决策层和前端都不需要额外修改。

### 4.3 VLM 结构化输出机制

感知层并不直接把 VLM 当作一个“自由描述机器人看到什么”的黑盒，而是通过 Prompt 约束其输出结构。系统中定义了默认场景任务提示词与视觉系统提示词，要求模型输出 JSON 对象，并显式包含 `scene_description`、`confidence` 和 `structured_observations` 三部分。其中，`structured_observations` 又必须至少包含 `objects`、`relations`、`robot_grasp_state` 和 `risk_flags` 等字段。通过这种方式，视觉输出不再是不可控的长段自然语言，而更接近一种受约束的结构化感知合同。

在实现上，不同 provider 的返回格式并不完全一致，因此系统内部还包含对 JSON 结果的提取与归一化逻辑。例如，对于 OpenAI 兼容视觉接口，会优先尝试使用 `response_format=json_object` 并解析返回内容；若解析失败，则退回到文本抽取逻辑。对于 Anthropic 兼容或 Ollama 兼容视觉接口，也分别做了适配与归一化处理。最终，无论底层 provider 差异如何，都会被转化为统一的 `SceneDescriptionResult` 结构。

### 4.4 主动感知与降级策略

真实环境中，感知并不总是稳定成功。相机可能暂时无输入，机器人状态适配器可能读取失败，VLM provider 也可能因为网络、鉴权或格式问题返回异常。为此，本文在感知层与决策层交界处设计了主动感知与降级机制。

首先，在 `assessment` 节点中，系统会根据 `perception_confidence` 判断当前感知结果是否达到阈值。如果置信度不足且主动感知次数未超限，则进入 `active_perception` 分支，提示系统重新聚焦目标物与风险区域后再次采集环境。其次，当视觉结果不可用，但任务本身属于关节归位或单舵机旋转这类更依赖机器人自身状态的操作时，系统允许进入 proprioceptive fallback，通过机器人关节状态构建简化场景摘要，避免因为视觉异常完全阻塞任务。

这意味着系统在论文中不应被表述为“视觉识别始终正确”，而应更真实地描述为：系统具备显式的感知置信度评估、主动感知重试和有限降级能力，在视觉不可用时会根据任务类型选择是否允许以关节状态闭环继续执行。

### 4.5 感知层当前实现状态

当前项目已经具备真实 VLM provider 装配能力，能够通过配置切换 `minimax_mcp_vision`、`openai_gpt4o`、`ollama_vision` 等感知模型来源，并保留 mock fallback 机制。同时，前端左上角实时视频流面板已能够在本地环境中显示来自摄像头的 MJPEG 流，为感知层运行状态提供直观证据。

当前系统已经形成“真实摄像头 + 真实 VLM provider 接入路径 + 结构化结果回传 + 视频流留痕”的完整感知链路，并能在前端工作台中展示感知结果与视频画面。

## 第5章 决策规划层设计与实现

### 5.1 决策层总体设计思路

决策层是系统中的“脑”，负责把自然语言任务、环境感知结果和执行反馈组织成一条可追踪、可验证、可回退的状态流。本文没有将决策过程简单理解为“让 LLM 直接决定下一步动作”，而是采用“状态图 + 模型规划 + 规则约束 + fallback”相结合的思路。其目标不是让模型拥有无限自由，而是在明确边界内利用模型增强自然语言理解和任务选择能力。

### 5.2 LangGraph 状态流转机制

决策层使用 LangGraph 构建了一张显式状态图。状态图中的主要节点包括：`trigger`、`nlu`、`sensory`、`assessment`、`active_perception`、`task_planning`、`pre_feedback`、`motion_control`、`verification`、`error_diagnosis`、`hri`、`compensation`、`success_notice`、`goal_check`、`state_compression` 和 `final_status`。每个节点都承担单一职责，并通过条件路由决定后续流向。

这种设计使系统运行过程具有良好的可解释性。开发者不仅可以通过代码理解状态流转逻辑，也可以在前端工作台中看到当前运行正处于哪一阶段、为什么进入下一阶段以及最终是如何收口的。在论文中，这一部分非常适合通过状态流图与 `run_id` 生命周期图配合说明，从而突出系统的“闭环控制”与“可观测性”特征。

### 5.3 任务规划与动作选择机制

当系统进入 `task_planning` 节点后，会综合用户原始指令、当前任务片段、场景描述和结构化观察结果，调用决策 provider 生成规划结果。规划输出被约束为结构化 JSON，其中包括 `selected_capability`、`selected_action`、`selected_action_args`、`reason` 以及面向用户的 `assistant_response`。在动作层面，系统支持把任务映射到 `describe_scene`、`servo_rotate`、`move_home`、`release` 或 `run_smolvla` 等动作空间中。

为了防止模型规划结果与任务实际语义不一致，系统还加入了显式的动作对齐约束。例如，当任务为单舵机旋转时，规划结果只能选择 `servo_rotate`；当任务为归位时，只能选择 `move_home`；当任务只是询问画面内容时，只能选择 `describe_scene`。如果模型输出与这些约束冲突，系统会自动覆盖为规则对齐后的动作，并在元数据中记录 `alignment_override`。这一设计是本文在工程上控制模型越权的一项重要措施。

### 5.4 Prompt 驱动的 JSON 规划约束

决策层中最核心的 Prompt 资产是 `DECISION_PLANNING_SYSTEM_PROMPT`。它要求模型只输出 JSON 对象，且字段必须围绕任务队列、能力选择、动作选择、动作参数、原因说明和用户回复组织。这一 Prompt 同时对舵机任务、归位任务和场景问答任务给出强约束，从源头减少模型把简单任务错误地规划为复杂抓取动作或 VLA 高阶动作的概率。

与单纯依赖模型“理解任务”不同，这种 Prompt 在系统中承担的是约束生成空间的角色。它与后续的动作对齐逻辑、fallback 规则和执行层安全边界共同构成多级防线。论文在描述这一机制时，可以强调 Prompt 工程并不是孤立技术，而是整个系统可控性设计的一部分。

### 5.5 汉诺塔任务 skill 扩展示例

为了展示系统面向复杂顺序任务的扩展潜力，本文在项目中新增了汉诺塔任务 skill。该 skill 并不直接替代现有执行链路，而是作为一个独立的任务技能资产存在。它能够识别汉诺塔类任务指令，依据“一次只能移动一个圆环”和“大圆环不能压在小圆环上方”等规则，递归生成三柱三环问题的合法步骤序列，并输出中文原子提示词列表。

这一设计有两个重要意义。第一，它证明当前系统不只是支持单步原子任务，还具备扩展到长程序列任务的能力。第二，它已经以专用系统提示词、skill card、规范文档和测试文件的形式沉淀为完整任务技能资产，因此完全可以作为论文中的“复杂任务技能扩展示例”进行讨论。

### 5.6 决策层当前实现状态

当前决策层已经实现真实 LLM provider 装配、LangGraph 状态流编排、Prompt 驱动 JSON 规划、动作对齐约束、heuristic fallback 与前端阶段事件可视化等能力。其运行结果能够通过 `assistant_response`、`selected_action`、`current_phase` 和 `eventFeed` 等字段直接回显到前端页面。

决策层当前已经完成“本地软件级闭环验证”和“真实 provider 接入可用”两类核心能力，并通过显式状态图与受限规划输出形成稳定的任务编排机制。

## 第6章 动作执行层设计与实现

### 6.1 执行层职责与工具集合

执行层承担的是“将规划结果真正变成动作请求并返回执行结果”的职责。本文中，执行层既包含底层机器人适配器，也包含上层工具注册、参数校验、安全管理和动作结果封套逻辑。目前系统已经注册的执行工具包括 `move_to`、`move_home`、`grasp`、`servo_rotate`、`release`、`clear_emergency_stop` 和 `run_smolvla`。

这些工具中，`move_to`、`move_home`、`grasp`、`servo_rotate` 与 `release` 可视为原子动作或近原子动作；`run_smolvla` 则可以视为高阶技能接口，允许更高层次地描述抓取与放置任务。执行层通过统一的 `MockMCPServer` 对外暴露工具定义，使决策层始终从“能力调用”的角度访问动作，而不是直接操作底层硬件接口。

### 6.2 原子动作与高阶技能边界

原子动作的优点在于可解释、可验证和可局部测试。例如，舵机旋转类任务可以明确地表达为 `servo_rotate(id, degrees)`，归位则可以表达为 `move_home()`。这些动作易于配合安全规则检查参数是否合法，也便于在失败时回溯问题原因。相比之下，高阶技能如 `run_smolvla` 更适合表达复杂的抓取流程，它将视觉输入、任务描述和机器人状态作为整体条件，生成若干内部步骤。

从当前项目完成边界来看，高阶技能已经不再停留在接口预留与 mock-first 验证阶段，而是完成了 SmolVLA 的实体执行接入。换言之，系统当前不仅实现了执行工具接口，还已经具备基于 SmolVLA 的真实动作执行能力。这使得执行层不再只是“可扩展设计”，而是已经进入“具备实体执行事实”的阶段。

### 6.3 安全检查与急停策略

机器人执行与纯软件系统最大的差异之一在于动作风险。即使规划结果在语义上正确，如果位移过大、舵机角度超限、抓取力不合理或遥测异常，仍可能带来硬件风险。为此，本文在执行层中引入了 `SafetyManager`，对动作请求进行前置安全检查和执行后安全校验。

具体而言，系统在执行前会检查机器人是否处于急停状态、连接状态是否正常、心跳是否正常以及是否存在硬件错误码。对不同类型动作，还会执行针对性的规则，例如 `move_to` 会检查单步位移阈值和奇异区边界，`grasp` 会检查力控上限，`servo_rotate` 会检查单次旋转幅度和目标角度是否超出安全范围。执行后，系统还会根据温度、电流、位置偏差等遥测指标判断当前动作是否处于安全区间。

这种多级安全链的意义在于，执行层不仅仅是“做动作”，更是系统最后一道物理风险防线。它与决策层的 Prompt 约束、动作对齐逻辑共同构成从“语言规划”到“物理动作”之间的安全桥梁。

### 6.4 遥测反馈与结果回传

执行层的输出并不是简单的成功或失败标志，而是包含动作名称、状态、消息、日志和附加元数据的结构化结果。这样设计的原因在于，机器人任务是否完成往往不止依赖动作是否被调用成功，还依赖执行后的状态是否满足预期。因此，执行结果必须能够被决策层用于下一步验证，也必须能够在前端界面中被观察和回放。

当前系统已经在运行态快照中展示 `last_execution`、`execution_feedback`、`robot_state` 和相关日志信息，使前端能够清楚地反映一次动作调用后的系统状态。这对于调试、答辩展示和论文留痕都具有直接价值。

### 6.5 执行层完成结果

当前执行层已经完成工具注册、安全边界、执行结果封套、统一适配器接口设计以及 SmolVLA 实体执行链路接入。这意味着本文现在可以明确写出：执行层已具备真实高阶技能执行事实，而不再局限于 mock execution 边界。

结合 `run_smolvla`、`move_home`、`servo_rotate` 等工具的接线与运行结果，执行层已经形成“动作注册 - 安全校验 - 适配器下发 - 遥测回传 - 结果封套”的完整执行闭环。

## 第7章 一体化工作台与后端服务实现

### 7.1 前端工作台设计目标

一体化工作台是本文系统的重要组成部分，它既是用户输入任务和查看结果的入口，也是系统可观测性建设的核心载体。与传统的命令行脚本相比，前端工作台能够在同一界面中展示当前任务输入、模型配置、工具能力、运行阶段、视频流画面、事件日志和终态反馈，从而极大提升系统调试与演示效率。

当前前端采用 React 19、TypeScript、Vite、Tailwind CSS 和 Zustand 构建，围绕运行态展示、配置面板、控制面板、工具面板和事件流面板组织页面结构。用户可以在右侧输入自然语言指令，点击“开始”提交任务，也可以在任务进行中点击“结束”请求停止当前运行。界面左上角展示实时视频流与遥测摘要，中部展示任务执行流与智能体回复，底部展示事件日志，从而形成较完整的运行态可视化面板。

### 7.2 后端 HTTP / SSE / 视频流服务设计

为了支持前端工作台的实时展示需求，后端提供了最小但完整的 HTTP 与 SSE 服务合同。`bootstrap` 接口用于页面初始化，返回配置、工具、执行模型和安全信息；`config` 接口用于读取和更新配置；`tools` 接口用于获取和刷新工具列表；`runs` 接口用于启动任务并返回 `run_id`、`snapshot_url` 和 `events_url`；`video-stream` 接口则以 MJPEG 形式输出当前视频流；`events` 接口通过 SSE 返回阶段事件。

这种设计使得前端能够以较稳定的方式消费运行时状态，而不必关心后端内部具体使用的是哪种 provider 或哪个适配器。当前后端以同步 WSGI + 进程内 `RunRegistry` 实现完成了运行任务、事件流和视频流的统一服务收口。

### 7.3 前端状态管理与接口接线

前端通过 Zustand 统一管理配置、工具、当前任务、快照、最新运行状态、事件列表和 SSE 连接对象。页面初始化时，会并行拉取 `bootstrap`、`config` 和 `tools`；用户提交任务后，前端通过 `POST /runs` 获得 `accepted.run` 与两个后续链接，然后建立 `EventSource` 订阅事件流，并在必要时通过 `snapshot_url` 做补偿同步；用户停止任务时，则调用 `snapshot_url/stop` 触发终态收口。

这种接线方式确保了页面不会仅依赖单一事件流，而是具备“事件订阅 + 快照补偿”的双重机制。实际联调中，这一点非常重要，因为运行过程可能由于事件流中断、浏览器重连或任务过快完成而导致界面错过部分状态更新。通过 `run_id` 生命周期与快照同步策略，页面能够更稳定地反映任务全过程。

### 7.4 `run_id` 生命周期管理

系统中的每一次运行任务都对应唯一的 `run_id`。后端通过 `RunRegistry` 为其维护事件列表、当前快照、终态标记、版本号与停止请求状态。前端则持有当前 `runAccepted` 信息，并根据事件流更新 `snapshot`、`latestRunState` 与 `eventFeed`。当任务进入终态后，事件源会被主动关闭，从而完成一次运行生命周期的收口。

这一设计既使任务过程具有可追踪性，也为论文中的时序图、实验记录和截图留痕提供了清晰结构。每次任务不仅有用户指令，还有唯一标识和对应状态序列，这对于系统验证和答辩解释都十分有利。

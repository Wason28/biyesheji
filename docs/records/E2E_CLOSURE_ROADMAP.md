# 端到端闭环推进路线图

## 文档定位

本文档用于整理项目从“当前本地 mock-first 可演示链路”推进到“Ubuntu 可复现链路”再到“真实模型 + 摄像头 + 机械臂实体闭环”的执行路线。

本文档不替代需求文档、阶段规格或测试报告；它的目标是给后续开发一个清晰的推进顺序，避免把“链路接上”误写成“项目完成”。

## 当前基线（2026-04-21）

当前已具备：

- 第一阶段骨架实现完成
- 第二阶段主线完成
- 第三阶段前端界面开发与 MCP 工具集成完成
- 本地最小闭环已可重复验收：
  - `frontend + backend/http + mock runtime` 可启动
  - `bootstrap / config / tools / runs / snapshot / events` 可用
  - 配置保存、工具刷新、run 提交、终态回放均可验证
  - 已有截图与 smoke 结果留档

当前仍不具备：

- Ubuntu 实机可复现链路
- 真实 `LLM / VLM / SmolVLA` 模型链路
- 真实摄像头输入与真实机器人状态读取
- 真实机械臂执行链路
- 实体分拣任务的重复成功记录

## 完成定义

### 1. 本地最小闭环完成

定义：

- 在本机代码环境中，能够稳定完成前端启动、后端启动、配置保存、工具刷新、run 提交、`snapshot/events` 回放与终态留痕
- 有脚本、截图、JSON 结果和记录文档可复现

说明：

- 当前项目已达到这一层级
- 这不是 Ubuntu 实机闭环，也不是实体闭环

### 2. Ubuntu 工程闭环完成

定义：

- 在 Ubuntu 目标环境中完成项目依赖安装、前后端联合启动与 smoke 验证
- 至少可重复完成一次“前端 -> 后端 -> 运行态 -> 终态”的完整链路
- 有 runbook、日志、截图和结果文件留档

说明：

- 这一层级证明部署链路可复现
- 仍不等于真实模型、真实摄像头和真实机械臂已完成接入

### 3. 真实实体闭环完成

定义：

- 接入真实 `LLM / VLM / SmolVLA`、真实摄像头、真实机械臂
- 用户指令能完成“感知 -> 决策 -> 执行 -> 验证 -> 前端回显”的真实链路
- 至少完成真实桌面分拣任务，并形成重复成功记录、失败分析和实验材料

说明：

- 这是项目层面的“真正闭环完成”口径
- 若用于毕业设计交付，还需同步完成论文与答辩材料

## 推荐推进顺序

## Phase 4A：本地最小闭环固化（已完成）

目标：

- 固化当前 mock-first 本地闭环，降低后续联调变量

已完成事实：

- `scripts/phase4_local_e2e_smoke.py`
- `docs/records/phase4_local_e2e_smoke_result_2026-04-21.json`
- `docs/records/phase3_workbench_2026-04-21.png`
- `docs/records/phase3_workbench_2026-04-21_live.png`

退出标准：

- smoke 可重复运行
- focused tests 通过
- 前端构建通过
- 文档与截图已留痕

## Phase 4B：Ubuntu 工程闭环

目标：

- 在 Ubuntu 上复现当前最小前后端链路

任务清单：

1. 准备 Ubuntu Python/Node/uv 环境
2. 安装项目依赖并记录版本
3. 启动 backend HTTP 服务
4. 启动 frontend dev server 或构建后静态服务
5. 跑 smoke 脚本并确认结果输出
6. 留存 Ubuntu 下日志、截图、产物 JSON
7. 更新 runbook 与测试记录

关键交付物：

- Ubuntu 运行截图
- Ubuntu smoke 结果 JSON
- Ubuntu 联合启动命令
- 故障排查说明

退出标准：

- Ubuntu 上能完整跑通一次最小闭环
- 文档中有明确复现步骤

## Phase 4C：真实感知链路接入

目标：

- 先把“输入侧”从 mock 替换为真实数据

任务清单：

1. 接入真实摄像头图像采集
2. 接入真实机器人状态读取
3. 验证 `get_image` / `get_robot_state` 输出结构稳定
4. 前端运行态正确显示真实图像与机器人状态
5. 补错误处理与超时边界

关键交付物：

- 真实图像输入截图
- 机器人状态读取记录
- 失败日志与恢复步骤

退出标准：

- 前端不再只显示 mock `current_image` 与 mock `robot_state`
- 真实输入能稳定进入决策链路

## Phase 4D：真实模型链路接入

目标：

- 让决策与感知不再依赖 mock provider

任务清单：

1. 接入真实 `LLM` provider
2. 接入真实 `VLM` provider
3. 验证配置面板可正确保存并切换模型参数
4. 验证 prompt/响应/超时/异常路径
5. 记录调用日志与失败样例

关键交付物：

- 模型配置留痕
- 真实模型调用成功记录
- 错误回退记录

退出标准：

- 至少一条完整 run 使用真实 `LLM/VLM`
- 错误路径可诊断

## Phase 4E：真实执行链路接入

目标：

- 把执行层从 mock 过渡到真实机械臂与真实 `SmolVLA`

建议顺序：

1. `move_home`
2. `move_to`
3. `grasp / release`
4. `run_smolvla`

任务清单：

1. 接入真实机械臂适配器
2. 接入真实 `SmolVLA` backend
3. 验证安全边界、急停、workspace 限制
4. 验证原子动作可控
5. 再验证完整技能调用

关键交付物：

- 回零成功记录
- 单步动作成功记录
- 安全策略说明
- 真实执行日志

退出标准：

- 真实执行链路可控且可回退
- 不因一次完整任务失败而丢失排障能力

## Phase 4F：真实实体闭环验证

目标：

- 完成真实桌面级分拣任务闭环

任务清单：

1. 选择固定桌面场景与目标物体
2. 运行完整指令闭环
3. 记录成功 / 失败 / 失败原因
4. 重复多轮实验
5. 统计成功率与失败分类
6. 回写测试、实验、图表和论文材料

关键交付物：

- 成功演示视频 / 截图
- 实验台账
- 成功率统计
- 失败分类图

退出标准：

- 真实任务不是只成功 1 次，而是有重复成功记录
- 有足够材料支撑论文与答辩

## 当前结论

从当前状态出发，项目已经具备“继续推进真实闭环”的工程基线。

最合理的下一步，不是直接宣告项目完整，而是按以下顺序推进：

1. Ubuntu 工程闭环
2. 真实感知链路
3. 真实模型链路
4. 真实执行链路
5. 实体分拣实验与统计

也就是说：

- 当前已完成“本地最小闭环”
- 下一阶段目标应是“Ubuntu 可复现闭环”
- 最终目标才是“真实实体闭环”

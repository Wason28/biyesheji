# 桌面级具身智能机器人感知-决策-执行一体化原型系统设计与实现

## 摘要

具身智能系统正在从单一算法研究走向感知、决策、执行协同的一体化系统研究。针对当前机器人系统常面临的模块耦合强、接口不统一、运行过程不可观测、前后端联调困难等问题，本文提出并实现了一套适用于桌面级具身智能机器人的“感知—决策—执行”三层解耦架构原型系统。该系统基于大语言模型（LLM）与视觉语言模型（VLM）构建，结合轻量视觉-语言-动作模型（SmolVLA），为桌面机械臂场景（如抓取、分拣）提供低成本、高可维护性、可观测的闭环控制能力。

本文详细设计了环境感知层、决策规划层和动作执行层。在环境感知层，实现了真实摄像头与虚拟 Mock 数据的双通道适配，通过统一感知合同对外提供结构化输出，并引入基于置信度评估的主动感知机制；在决策规划层，基于 LangGraph 构建了有状态、可回退的决策编排流，引入多 Agent 协作范式，完成了对多款 LLM（如 Minimax、OpenAI、Ollama）的适配与基于启发式的任务拆解回退能力，最终形成闭环反馈；在动作执行层，统一定义了原子动作与高阶技能工具（如 `move_to`、`grasp`、`run_smolvla`），封装了 SmolVLA 模型能力，并在执行前后设计了严格的多级安全检查（如位移阈值限制与奇异点规避）与状态遥测反馈机制。

原型系统前端基于 React 19 构建了一体化运行工作台，后端通过最小 WSGI HTTP 服务提供状态流转与事件推送（SSE）能力。实验验证表明，本系统在本地软件级已实现最小闭环验证，各项核心测试（覆盖 51 项后端单元/集成测试与前端 Playwright E2E 回归测试）均稳定通过；同时，执行层的 SmolVLA 实体执行链路已经完成打通，并已产出汉诺塔示教数据上的 SmolVLA 训练报告、训练曲线与日志材料。该系统已经形成一套可运行、可观测、可留痕的桌面级具身智能原型。

**关键词**：具身智能，感知-决策-执行解耦，大语言模型（LLM），视觉语言模型（VLM），LangGraph，多智能体，安全机制

## 第1章 绪论

### 1.1 研究背景

随着深度学习与人工智能技术的迅猛发展，机器人领域正在经历从“程序化自动化”向“认知型具身智能（Embodied AI）”的范式转变。传统的机器人控制系统通常高度依赖人工设定的规则和精确的环境建模，在面临开放世界（Open-world）的动态环境时，常表现出脆弱性和低泛化性，这也被称为“莫拉维克悖论”（Moravec's paradox）。近年来，大语言模型（Large Language Models, LLMs）和视觉语言模型（Vision-Language Models, VLMs）在常识推理、语义理解和跨模态映射上展现出了惊人的涌现能力（Emergent Abilities）。将这些基础模型作为机器人的“大脑”进行任务规划，作为“小脑”进行直觉运动控制，已成为当前学术界和工业界的研究热点。

然而，将大模型引入机器人控制链路时，面临着严峻的工程挑战。首先，感知、决策与执行模块之间往往存在极强的代码耦合，导致系统难以复用与扩展；其次，大模型的输出具有非确定性和幻觉（Hallucination），缺乏严格的物理边界与安全约束；最后，传统的机器人操作系统（如 ROS）学习曲线陡峭，且缺乏现代化、高可观测性的调试前端，使得研究人员在进行“算法-硬件”联调时面临“黑盒化”的困境。桌面级机械臂凭借其成本低、任务定义清晰（如抓取、堆叠、分拣）、便于实验复现等特点，逐渐成为验证具身大模型能力的理想平台。

### 1.2 研究意义

针对上述挑战，本文提出一套“感知—决策—执行”解耦架构的系统方案。通过统一的合同接口（Capability Contract）、状态流转（State Graph）和直观的前端工作台，大幅提升系统的可维护性、可扩展性和可验证性。本系统已经完成本地软件级闭环验证、SmolVLA 实体执行链路打通、前端工作台联调和训练材料沉淀，具有明确的学术探索价值和工程实践意义。

### 1.3 国内外研究现状

在具身智能的决策架构方面，Google 的 SayCan、RT-1、RT-2 奠定了基于语言指令和视觉输入的端到端或分层控制基础；Stanford 提出的 VoxPoser 证明了 LLM 能够生成具有物理意义的价值映射以指导动作。在系统编排层面，LangChain 及其扩展框架 LangGraph 逐渐被用于构建有状态的多智能体（Multi-Agent）协作流。然而，多数开源项目要么侧重于纯算法端的离线评估，要么直接绑定特定的昂贵硬件，鲜有提供一套完整、解耦且带有现代化 Web 监控工作台的开源系统范式。

### 1.4 论文结构组织

本文围绕系统“感知-决策-执行”的三层解耦架构展开详细论述：

## 第2章 相关技术与理论基础

### 2.1 具身智能与多模态大模型技术

具身智能强调智能体通过物理实体与环境进行交互来获得认知。在这一过程中，多模态大模型（如 GPT-4o、Minimax、Ollama 部署的本地模型）充当了语义到物理动作的桥梁。视觉-语言-动作模型（VLA, Vision-Language-Action Models），如本系统引入的 SmolVLA，是一种轻量级的端到端网络，能够直接将像素和指令映射为机械臂的关节位移或笛卡尔坐标。这类模型依赖于 Transformer 架构，通过对海量人类示范轨迹和视觉数据的交叉注意力（Cross-Attention）训练，实现了基础的直觉物理常识。

### 2.2 机器人运动学理论基础

机器人的动作执行依赖于严格的运动学（Kinematics）理论。对于桌面级机械臂，其末端执行器（End-effector）在三维笛卡尔空间中的位姿表示为 $P = (x, y, z, roll, pitch, yaw)$。
系统将底层关节空间映射到任务空间，需使用正向运动学（Forward Kinematics）。采用标准的 D-H（Denavit-Hartenberg）参数法，相邻连杆 $i-1$ 到 $i$ 的齐次变换矩阵 $T_{i}^{i-1}$ 可表示为：
$$ T_{i}^{i-1} = \text{Rot}_z(\theta_i) \text{Trans}_z(d_i) \text{Trans}_x(a_i) \text{Rot}_x(\alpha_i) $$
其中 $\theta_i$ 为关节角，$d_i$ 为偏置，$a_i$ 为连杆长度，$\alpha_i$ 为扭转角。系统在 `SafetyManager` 中执行空间边界校验时，大量依赖于这类变换求得的末端绝对坐标。

### 2.3 状态机与有向无环图 (DAG) 理论

为了管理复杂的决策与容错流转，系统采用了基于有向无环图（DAG）的有限状态机（Finite State Machine, FSM）。在形式化定义中，该状态机可以表示为一个五元组：
$$ M = \langle S, s_0, \Sigma, \delta, F \rangle $$
其中：
- $S$ 为状态集合（如 `sensory`, `task_planning`, `motion_control` 等节点）。
- $s_0$ 为初始状态（`trigger`）。
- $\Sigma$ 为输入字母表（如用户的自然语言指令、感知反馈）。
- $\delta: S \times \Sigma \rightarrow S$ 为状态转移函数（通过 LangGraph 的 `add_conditional_edges` 实现）。
- $F$ 为终止状态集合（`final_status`, `END`）。
这种数学模型确保了系统在遇到感知不确定性或执行失败时，能够有界且确定地跳转至重试或人类干预节点，避免死锁。

### 2.4 前后端分离架构与流式通信

在监控与调试体系中，实时性至关重要。传统的 HTTP 请求/响应模型无法满足机器人遥测数据的低延迟推送需求。本系统采用 Server-Sent Events (SSE) 协议，基于 HTTP 建立单向长链接。服务器可以持续推送 `DecisionAgentState` 的微小增量（如当前节点变化、遥测坐标更新），前端 React 状态管理库 Zustand 接收并合并这些增量，从而实现 60FPS 级别的状态回放与日志刷新。

## 第3章 系统总体设计与架构

### 3.1 系统需求分析

桌面级具身智能研究系统需要满足以下核心需求：
1. **模块解耦**：感知、决策、执行三大模块必须具备独立替换的能力（如将 Mock 相机无缝切换为 Real 相机）。
2. **安全可控**：大模型的输出具有不可预测性，必须在执行层设置硬性的物理边界防护。
3. **高可观测性**：研究人员需要直观地看到大模型当前的推理节点、感知到的画面以及底层关节数据。
4. **容错与闭环**：在动作失败时，系统应具备自动重试或寻求帮助的能力，而非直接崩溃。

### 3.2 总体架构设计

基于上述需求，本系统设计了“感知-决策-执行”三层解耦架构，并在外围辅以“一体化工作台”。

[图 1：系统总体架构图]

系统架构自下而上包含：
- **执行层 (Execution Layer)**：封装硬件驱动与基础控制算法（如 `move_to`, `grasp`），包含严格的 `SafetyManager` 与 `SmolVLA` 推理引擎。
- **感知层 (Perception Layer)**：提供摄像头驱动抽象（`CameraAdapter`）与 VLM 语义抽取接口，产出标准化的场景快照。
- **决策层 (Decision Layer)**：系统核心枢纽，基于 LangGraph 构建。负责任务拆解、状态机编排与执行结果验证。
- **控制台与接口层 (Workbench & API)**：通过最小 WSGI HTTP/SSE 服务暴露运行时接口，通过 React 19 提供交互式 GUI 面板。

### 3.3 接口与合同规范设计

为了彻底解耦，层与层之间的通信不使用自由格式的字典，而是定义了强类型的“能力合同”（Capability Contracts）。例如，决策层不关心相机是如何获取图像的，它只调用 `describe_scene` 接口，并期待返回符合 `RobotStateSnapshot` 和 `CapturedImage` Schema 的结构化 Pydantic 数据类。这种面向契约（Contract-Oriented）的编程范式是系统保持高可维护性的关键。

## 第4章 环境感知层设计与实现

环境感知层是具身智能系统与物理世界交互的第一入口，其任务是将非结构化的物理信号（光子、电流）转化为决策层可理解的结构化语义。

### 4.1 视觉感知通道设计

系统在 `src/embodied_agent/perception/adapters.py` 中实现了多种感知通道的抽象：
- **真实硬件通道 (`OpenCVCamera`)**：支持动态帧率（FPS）、分辨率缩放和流媒体编码，通过 V4L2 或 DirectShow 获取底层 USB 摄像头数据。
- **虚拟通道 (`MockCamera`)**：在缺乏硬件时，读取本地测试图像并转化为 Base64 流，保证开发测试不中断。
- **流媒体推送 (`iter_mjpeg_stream`)**：通过生成器模式，将原始图像帧封装为 multipart MJPEG 数据流，供前端实时展示。

### 4.2 多模态大模型适配器

视觉信息需要转化为语言才能被 LLM 处理。系统在感知层集成了统一的 Provider 工厂（`build_perception_provider`）。针对不同的部署环境，支持：
1. **MCP 接入**：通过 Model Context Protocol 接入 `minimax_mcp_vision` 等前沿模型。
2. **直接 API 接入**：通过标准 HTTP 接入 `openai_gpt4o` 或本地化部署的 `ollama_vision`。
系统内置了精心设计的 System Prompt，引导 VLM 提取场景中的物体类别、颜色、相对空间方位以及环境遮挡状态。

### 4.3 结构化数据合同

感知模块严格遵循标准化合同，避免向上传递无用日志：
- **CapturedImage**：包含 Base64 图像、UTC 时间戳、分辨率（width, height）以及相机的内参/外参配置（如 $f_x, f_y, c_x, c_y$）。
- **RobotStateSnapshot**：包含机械臂各关节角度、末端执行器的笛卡尔位姿以及夹爪的开合状态。

[表 1：感知层核心合同字段定义]

### 4.4 主动感知与置信度评估算法

针对复杂环境下的遮挡与模糊问题，系统在 LangGraph 中特设了 `assessment`（状态置信度评估）节点。其置信度 $C$ 的计算可抽象为多源信息的加权评估：
$$ C = \sum_{i} w_i \cdot p_i $$
其中 $p_i$ 代表 VLM 对特定特征（如目标物体轮廓、指令匹配度）的概率评估，$w_i$ 为权重。当 $C < \tau$（$\tau$ 为系统预设阈值）时，状态机将自动路由至 `active_perception` 节点，主动微调机械臂姿态以获取新视角图像，补全感知盲区后再进入规划阶段。

## 第5章 决策规划层设计与实现

决策层是系统的“大脑”，负责接收指令、拆解任务并监控执行结果。系统在 `src/embodied_agent/decision/graph.py` 中实现了复杂的 DAG 流转。

### 5.1 基于 LangGraph 的状态机编排

系统将决策过程抽象为 16 个核心节点的有向无环图，状态对象定义为 `DecisionAgentState`：
1. **Trigger & NLU**：接收前端指令输入并解析意图。
2. **Sensory & Assessment**：调用感知层并评估置信度，决定是否触发 Active Perception。
3. **Task Planning**：将抽象意图拆解为具体动作步骤列表。
4. **Motion Control & Verification**：将单步动作下发至执行层，随后通过遥测数据验证动作是否成功。
5. **Error Diagnosis & Compensation**：在验证失败时诊断错误类型，并生成补偿动作。
6. **Goal Check & Final Status**：判断整体目标是否达成，更新上下文记忆。

[图 2：LangGraph 决策流转状态机拓扑图]

### 5.2 多 Agent 协同机制

在上述流程中，采用了“术业有专攻”的多 Agent 协作范式：
- **规划Agent**（`task_planning_node`）：负责生成 `move_to`, `grasp` 等工具调用序列。
- **验证Agent**（`verification_node`）：仅负责比对预期坐标与实际遥测坐标。
- **诊断Agent**（`error_diagnosis_node`）：在触发安全异常（如碰撞）时，分析底层抛出的 `SafetyError` 日志，向规划层反馈人类可读的修复建议。

### 5.3 层次化任务分解与启发式回退

具身任务指令通常具有高度抽象性。系统采用宏观-微观结合的层次化分解策略。
为防止大模型 API 故障导致系统停摆，决策引擎内置了启发式回退（Heuristic Fallback）机制。当 LLM 响应超时或格式错误时，系统回退到基于规则的正则匹配算法，依靠硬编码状态机（如“发现抓取关键字 -> 生成移动与抓取指令”）完成最低限度的可用性保障（Minimum Viable Capability）。

### 5.4 闭环反馈与误差补偿

闭环反馈是防止物理任务灾难的关键。在 `verification_node` 中，如果目标坐标为 $P_{target}$，实际到达坐标为 $P_{actual}$，系统计算空间欧氏距离误差。若误差大于允许容差，状态机将流转至 `compensation` 节点，LLM 将基于当前的残差重新规划微调动作。若重试次数超过 `max_iterations`，则转入 `hri`（人机交互）节点，等待人工接管。

## 第6章 动作执行层设计与实现

执行层（位于 `src/embodied_agent/execution/`）是系统的“小脑”和“肢体”，负责物理动作的落地与安全保障。

### 6.1 统一执行工具定义

系统暴露了严格规范的原子动作工具：
- `move_to(x, y, z, roll, pitch, yaw)`：规划笛卡尔空间轨迹。
- `move_home()`：沿安全轨迹返回零点。
- `grasp(force)` & `release()`：控制夹爪。
所有工具调用均返回 `ExecutionToolResult`，强制包含执行状态、步骤日志及最新的 `RobotState`，形成闭环数据流。

### 6.2 轻量级 VLA 模型 (SmolVLA) 封装

针对非结构化抓取任务，系统集成了 SmolVLA 模型工具 `run_smolvla`。该工具不仅接收位姿指令，还接收当前视角的图像。在后端，模型将图像特征提取后与文本指令特征融合，通过自回归生成动作 Token 序列，并解码为底层的位姿增量。这种设计赋予了系统强大的零样本泛化能力。结合当前项目进展，`run_smolvla` 已不再停留在接口占位或纯 mock 验证阶段，而是已经完成实体执行链路接入，可作为执行层高阶技能的真实实现依据。

### 6.3 严格的多级安全防护机制

具身安全是系统设计的红线。在 `src/embodied_agent/execution/safety.py` 中，`SafetyManager` 实现了严格的数学边界防护：
1. **输入校验 (Input Validation)**：防止越界参数注入。
2. **执行前检查 (Preflight Check)**：
   计算目标位置相对于当前位置的单步位移距离：
   $$ d = \sqrt{(x_t - x_c)^2 + (y_t - y_c)^2 + (z_t - z_c)^2} $$
   如果 $d > D_{max}$（例如 0.1m），则触发 `SafetyError` 拦截。
   同时进行奇异点规避检查，若 $|z| < Z_{singularity}$，拒绝执行以防止电机过载。
3. **遥测验证 (Telemetry Check)**：实时监控电机温度、电流与跟随误差。

### 6.4 遥测反馈链路

每一次执行动作后，底层适配器会通过高速总线读取关节的编码器反馈。这些数据经过重采样和滤波后，封装进 `RobotStateSnapshot` 返回决策层，并同步推送至前端的 Event Panel，形成了从指令下发到状态确认的完整闭环。

## 第7章 一体化工作台与后端服务实现

为了打破传统机器人开发的黑盒状态，本研究开发了现代化的前后端系统。

### 7.1 后端服务框架与流式推送

后端基于 Python 最小 WSGI HTTP 传输层构建，在轻量化原型条件下提供统一的运行时 API。除了提供常规的 RESTful 风格接口用于配置管理和任务提交外，其核心亮点在于 SSE 事件流引擎。
在 LangGraph 每执行完一个节点，系统会触发一个 `RunEvent`。SSE 引擎将该事件序列化为 JSON，通过持久连接推送到前端，避免了短轮询带来的延迟与服务器压力。

### 7.2 前端状态管理与面板设计

前端工作台（`biyesheji/frontend/`）基于 React 19 和 TypeScript 构建，采用 Zustand 进行无中心化的状态流转。
界面布局划分为多个核心组件：
- **Runtime Panel**：展示当前任务状态、节点时序与执行耗时。
- **Control Panel**：提供任务启动、任务结束、自然语言输入与运行状态概览。
- **Event Panel**：以流式日志的形式滚动展示感知结果、决策推理过程与底层安全拦截信息。
- **Vision Feed**：实时渲染来自后端的 MJPEG 视频流与覆盖于其上的语义包围盒。
